# Update Spec

Hyperfocus의 4개 문서는 한 덩어리다:

- `docs/spec.md` — 무엇을 (요구사항)
- `docs/architecture.md` — 어떻게 (구조)
- `docs/plan.md` — 어떤 순서로 (구현 단계)
- `CLAUDE.md` — 가벼운 색인

기능 변경은 거의 항상 4개 모두에 영향을 끼친다. 한두 개만 손대면 곧 불일치가 쌓이고, 다음에 코드를 여는 Claude가 어느 문서를 믿어야 할지 모르게 된다. 이 스킬의 목적은 그 불일치를 원천 차단하는 것이다.

## 핵심 원칙

### 1. 통합(integration) > 추가(bolt-on)

새 동작을 기존 구조에 자연스럽게 녹여라. "맨 뒤에 새 섹션 하나 추가하고 끝"은 거의 항상 잘못된 신호다.

**잘못된 예 (bolt-on):**

- 사용자: "전체 리셋에 비밀번호 보호 추가하자"
- 잘못된 반응: SPEC §12 "비밀번호 보호" 새로 만들고 끝.

**올바른 예 (integrated):**

- §5.4 (종료 동작) 안에 비밀번호 확인 단계를 흐름의 일부로 포함
- §10 엣지 케이스 표에 "비밀번호를 잊은 경우" 행 추가
- §11 Out of Scope에 모순되는 항목이 있다면 정리
- architecture §6 (저장 스키마)에 비밀번호 해시 필드 추가, §1.4 영속화 절에 평문 미저장 명시
- plan의 관련 Phase에 검증 항목 추가

새로운 도메인 개념(예: "비밀번호")이 등장하면 SPEC §2 용어 정의에 먼저 추가하고 본문에서 일관되게 그 용어를 쓴다.

### 2. 교체(replace) > 공존(coexist) — 충돌 시 반드시 구 내용 삭제

새 내용이 기존 내용과 충돌할 때, **기존 내용을 지우고 새 내용으로 교체**한다. 충돌하는 두 내용이 문서에 동시에 존재하는 상태는 절대 허용하지 않는다.

**잘못된 예 (공존):**

- 사용자: "리셋 후 타이머는 0이 아니라 일시정지 상태로 시작하자"
- 잘못된 반응: §5.2에 "리셋 후 0으로 초기화" 문장을 그대로 두고, 아래에 "단, 일시정지 상태로 시작할 수도 있음" 문장을 추가.

**올바른 예 (교체):**

- §5.2의 기존 "리셋 후 0으로 초기화" 문장을 **삭제**하고 "리셋 후 0으로 초기화, 일시정지 상태" 문장으로 교체.
- §10 엣지 케이스 표에서 기존 동작을 전제한 행도 함께 갱신.

**점검 기준:** 문서를 처음 읽는 사람이 "두 설명 중 어느 것이 맞지?"라는 의문을 품을 수 있다면, 그건 충돌이 남아있는 것이다. 수정 후 각 문서를 처음부터 끝까지 읽으며 모순이 없는지 확인한다.

> **이 원칙은 Step 8 최종 일관성 점검에서도 명시적으로 체크된다.**

### 3. cascade 순서 엄수

항상 이 순서:

```
1. docs/spec.md          ← 사용자가 보는 것이 먼저 결정됨
2. docs/architecture.md  ← SPEC 결정에서 파생됨
3. docs/plan.md          ← SPEC + architecture 결과를 검증하는 계획
4. CLAUDE.md             ← 영향 있을 때만 (대부분 손 안 댐)
```

순서를 바꾸면 (예: plan 먼저, SPEC 나중) 일관성 검증이 무의미해진다.

### 4. 먼저 전부 읽고, 그 다음에 손대기

수정 시작 전에 4개 문서를 처음부터 끝까지 모두 Read한다. "관련 섹션만 찍어서" 읽지 않는다. 한 변경은 보통 여러 섹션과 얽혀있고, 어디가 얽혀있는지 미리 파악해야 한다.

## 워크플로

### Step 1 — 4개 문서 전부 읽기

`docs/spec.md`, `docs/architecture.md`, `docs/plan.md`, `CLAUDE.md`를 빠짐없이 Read한다.

### Step 2 — 변경 의도 명확화

사용자가 자연어로 한 설명을 다음으로 정리한다:

- **변경 유형**: 추가 / 수정 / 삭제
- **핵심 동작**: 한 문장 요약
- **사용자 경험 변화**: 사용자가 무엇을 다르게 느끼는가

다음 중 하나라도 애매하면 AskUserQuestion으로 물어본다:

- 기존 §11 Out of Scope와 충돌 가능성이 있는가
- 데이터 스키마 변경이 동반되는가
- 기존 UI 요소를 대체하는가, 추가되는가
- 같은 영역에 이미 존재하는 동작과 동시에 가능한가, 배타적인가

### Step 3 — 영향 범위 매핑

수정에 앞서, 어디를 손대야 하는지 목록을 만든다. 변경 규모가 작으면(예: 한 라벨 텍스트) 머릿속에서만 해도 되고, 중규모 이상이면 사용자에게 보여주고 한 번 합의받는다.

체크 항목 (모든 변경에 적용 가능한 표준 체크리스트):

```
spec.md
  □ §2 용어 정의: 새 용어 등장? 기존 용어 의미 변경?
  □ §3 메뉴바 동작: 아이콘/라벨 동작 변경?
  □ §4 Popover 구성: 화면 요소 추가/삭제/재배치?
  □ §5 타이머 동작: 액션의 의미/순서 변경?
  □ §6 Title 규칙: 입력/표기/합산 규칙 영향?
  □ §7 통계 규칙: 기록/표시/계산 방식 변경?
  □ §8 슬립 처리: 자동 동작 추가/변경?
  □ §9 데이터 보존: 복원 동작 영향?
  □ §10 엣지 케이스 표: 행 추가/수정 (거의 항상 발생)
  □ §11 Out of Scope: 모순 항목 정리

architecture.md
  □ §1 기술 스택: 새 프레임워크/시스템 이벤트?
  □ §3 폴더 트리: 파일 추가/이동/삭제
  □ §3.1 책임 표: 새 컴포넌트 책임?
  □ §3.2 의존 방향: 변경?
  □ §4 컴포넌트 책임 상세: 액션/상태 추가?
  □ §5 데이터 흐름 시나리오: 새 시나리오 추가?
  □ §6 저장 스키마: 필드 추가/삭제?
  □ §7 테스트 전략: 새 테스트 영역?
  □ §8 변경 영향 가이드: 새 entry 추가 (자주 누락됨!)

plan.md
  □ 이미 완료된 Phase의 할 일 갱신?
  □ Phase에 검증 항목 추가?
  □ 새 Phase 신설?
  □ 단계 의존 관계 다이어그램 갱신?
  □ 완료 기준 추가?

CLAUDE.md
  □ 빠른 참조 갱신 (저장 경로, 진입점, 타깃 OS 등)?
  □ 작업 규칙에 새 원칙 추가?
  □ 새 문서 링크?  (보통 N)
```

### Step 4 — spec.md 수정

매핑에 따라 수정한다. 다음 원칙을 지킨다:

- 새 동작이 기존 섹션의 자연스러운 확장이면 **그 섹션 안에서** 처리한다.
- 정말 새로운 도메인 개념이면 **§2 용어 정의에 먼저 추가**하고 본문에서 일관되게 쓴다.
- 엣지 케이스는 **§10 표에 행 단위로** 추가한다. 별도 섹션으로 빼지 않는다.
- Out of Scope에 있던 항목을 구현하기로 했다면 **§11에서 명시적으로 제거**한다 (잊으면 다음 회차가 헷갈린다).
- 반대로 새 비-목표가 생겼다면 §11에 추가한다.

수정이 끝나면 spec.md 내부 정합성을 점검한다:

- §2 용어가 본문에서 다른 단어로 불리고 있지는 않은가?
- §10 엣지 케이스가 §5 동작과 모순되지 않는가?
- §11 Out of Scope에 본문이 구현 중인 항목이 남아있지는 않은가?

### Step 5 — architecture.md 수정

업데이트된 SPEC.md를 다시 한 번 빠르게 훑고, 그것을 기반으로 architecture.md를 수정한다:

- 새 기능이 기존 폴더 트리 안에서 처리 가능하면 **새 폴더/파일을 만들지 말고** 기존 위치에 넣는다.
- 정말 새 레이어/컴포넌트가 필요하면 §3 폴더 트리, §3.1 책임 표, §3.2 의존 방향을 **함께** 수정한다 (하나만 고치면 가이드가 무너진다).
- 데이터 변경은 §6 스키마와 §1.4 영속화 절을 **함께** 갱신한다.
- 새 시스템 이벤트(슬립, 알림, 키보드 등)는 §1.5에 추가한다.
- 새 사용자 액션이나 시스템 트리거는 §5 데이터 흐름 시나리오에 한 절 추가한다.
- **§8 변경 영향 가이드** 표를 잊지 말 것 — 새 기능 영역이 생겼는데 이 표에 entry를 안 만드는 건 가장 흔한 누락이다.

### Step 6 — plan.md 수정

- 이미 완료된 Phase에 속하는 변경이면 그 Phase에 **할 일/검증을 추가**한다. 단계 번호는 유지.
- 기존 Phase로 흡수 불가능한 큼직한 변경이면 **새 Phase를 신설**한다. 의존 관계상 올바른 위치에 끼워넣는다.
- 새 Phase가 추가됐다면 **단계 의존 관계 다이어그램을 갱신**한다.
- 완료 기준(Definition of Done)에 새 항목 추가가 필요한지 확인한다.

### Step 7 — CLAUDE.md 검토

CLAUDE.md는 의도적으로 가볍게 유지된다. 다음 경우에만 손댄다:

- 빠른 참조의 사실이 부정확해진 경우 (저장 경로, 진입점, 타깃 OS 등).
- 작업 규칙에 새 원칙이 정말로 추가되어야 하는 경우.
- 새 문서가 추가된 경우 (드물다).

대부분의 기능 변경은 CLAUDE.md를 안 건드린다. 손대고 싶은 충동이 들면 한번 더 의심하라.

### Step 8 — 최종 일관성 점검

4개 문서 수정이 끝났으면 한 바퀴 더 돌면서 확인한다:

- SPEC에 정의된 새 동작이 plan의 어떤 Phase에서 검증되는가?
- architecture에서 명명된 새 파일/컴포넌트가 SPEC의 어떤 요구사항을 충족시키는지 추적 가능한가?
- plan 의존 관계 다이어그램이 새 Phase를 반영하는가?
- CLAUDE.md의 빠른 참조가 여전히 정확한가?
- 4개 문서에서 같은 개념이 서로 다른 단어로 불리고 있지는 않은가?
- **변경된 내용과 충돌하는 기존 서술이 어느 문서에도 남아있지 않은가?** (핵심 원칙 §2 교체 원칙) 처음 읽는 사람이 "어느 설명이 맞지?"라고 헷갈릴 여지가 있으면 즉시 수정한다.

발견된 불일치는 즉시 수정한다.

### Step 9 — 코드 작성

문서가 확정되면 변경 내용을 실제 코드에 반영한다:

- architecture.md에 명시된 파일 위치에 정확히 코드를 배치한다.
- CLAUDE.md의 코드 규칙을 따른다: 평범한 Swift, `os.Logger` 로깅, 에러를 삼키지 않기.
- 임시 코드·주석 처리된 블록·실험 코드를 남기지 않는다.
- 코드 작성 전후로 빌드 경고 0개를 유지한다.

### Step 10 — 테스트

- 도메인 로직(Stores, 합산, 영속화) 변경이 있으면 단위 테스트를 작성 또는 갱신한다.
- `xcodebuild test`로 전체 테스트 스위트를 실행해 회귀가 없는지 확인한다.
- 빌드 에러·경고·테스트 실패가 있으면 즉시 수정한 뒤 다시 통과시킨다.
- 테스트를 통과한 상태에서만 Step 11로 넘어간다.

### Step 11 — 결과 보고

사용자가 요청한 내용이 어떻게 처리됐는지 **자연어**로 보고한다.

보고 원칙:

- 사용자의 요청을 기준으로 "무엇이 어떻게 바뀌었는가"를 설명한다.
- MD 파일 수정 목록, 코드 diff, 섹션 번호 나열은 하지 않는다.
- 테스트를 실행했다면 통과 여부를 한 줄로 언급한다.
- 후속으로 사용자 결정이 필요한 사항이 있다면 마지막에 별도로 언급한다.

예시 (좋은 보고):

> 전체 리셋 버튼을 누르면 이제 비밀번호를 먼저 확인합니다. 비밀번호가 틀리면 리셋이 진행되지 않고, 3회 연속 실패 시 1분간 잠깁니다. 기존 52개 테스트 모두 통과했습니다.

예시 (나쁜 보고):

> - docs/spec.md: §5.4 수정, §10 행 2개 추가
> - docs/architecture.md: §6 필드 추가
> - HyperfocusStore.swift: resetAll() 수정
> - HyperfocusStoreTests.swift: 테스트 3개 추가

## 변경 규모별 가이드

### 소규모 — 라벨 변경, 단일 엣지 케이스 추가, 단순 문구 조정

- Step 3 영향 범위 매핑을 사용자에게 보여주는 것 생략 가능
- 보통 SPEC만 수정되고 architecture는 무변, plan은 한 줄 추가 정도, CLAUDE는 무변
- 코드 변경이 없거나 미미하면 Step 9/10도 빠르게 끝남

### 중규모 — 새 UI 요소, 새 입력 필드, 동작 흐름 변경

- Step 3 매핑을 사용자에게 보여주고 한 번 합의받기
- architecture §3 폴더 트리에 파일 추가 가능성 있음
- plan 기존 Phase에 할 일 추가
- 단위 테스트 추가 가능성 높음

### 대규모 — 새 화면, 데이터 모델 변경, 새 시스템 통합

- Step 3 매핑 + 새 데이터 흐름 시나리오 초안을 미리 보여주고 합의받기
- architecture §3/§4/§5/§6/§8 모두 갱신 필요할 가능성 높음
- plan에 새 Phase 신설 가능성 높음 — 의존 관계 다이어그램 반드시 갱신
- 데이터 모델 변경이면 schemaVersion 증가 및 마이그레이션 정책 명시 검토
- 단위 테스트 대폭 추가 필요

| 함정                                  | 방지법                                                                                                                                                                         |
| ------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **충돌 내용 공존 — 가장 흔한 실수**   | 새 동작이 기존 서술을 대체할 때, 기존 서술을 **반드시 삭제**한다. 추가만 하고 지우지 않으면 문서가 모순을 품게 된다 — Step 8의 교체 원칙 점검을 통과하지 못하면 수정 완료 아님 |

## 삭제 변경에 대한 주의

"X 기능 빼자" 같은 삭제 요청은 추가/수정보다 위험하다:

- 삭제 대상이 다른 기능의 전제(precondition)가 아닌지 확인한다 (예: "Title" 삭제 → 통계 합산 규칙 §6.3이 의미를 잃음).
- 단순히 SPEC 섹션 한두 개를 지우지 않는다. 그 기능에 의존하던 §10 엣지 케이스, §7 통계 규칙, architecture의 컴포넌트, plan의 Phase까지 모두 함께 정리한다.
- §11 Out of Scope에 명시적으로 이동시키는 것을 고려한다 ("이번 버전에서는 다루지 않음"으로 의도를 남김).
- 데이터 스키마에서 필드를 제거하는 경우 마이그레이션이 필요한지 사용자에게 확인한다.
- 코드에서도 해당 기능을 완전히 제거한다 — 사용되지 않는 코드를 남기지 않는다.

## 이 스킬을 쓰지 않는 경우

- 단순 오타 수정, 문법 다듬기는 cascade 불필요.
- 코드만 고치고 문서 영향이 없는 리팩토링.
- 사용자가 명시적으로 "한 문서만 보고 싶다"고 말한 경우.

위 경우 외에는 거의 항상 이 스킬이 발동되어야 한다. 의심스러우면 발동시키고, Step 3 영향 범위 매핑에서 "변경 없음"으로 빠르게 결론지으면 된다.
