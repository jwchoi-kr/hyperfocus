# Hyperfocus

macOS 메뉴바 스탑워치 앱. 두 개의 타이머(현재 세션 / 전체)로 작업 시간을 기록하고,
주기별로 세션별 작업 시간 통계를 누적한다.

## 문서

작업 전에 관련 문서를 먼저 읽고, 의문이 생기면 코드보다 문서를 1차 기준으로 삼는다.

- [docs/SPEC.md](./docs/SPEC.md) — **무엇을** 만드는지. 기능 요구사항, UI 동작, 엣지 케이스, Out of Scope.
- [docs/architecture.md](./docs/architecture.md) — **어떻게** 만드는지. 기술 스택, 폴더 구조, 레이어 책임, 데이터 흐름.
- [docs/plan.md](./docs/plan.md) — **어떤 순서로** 만드는지. Phase 0~8, 각 단계의 검증 기준.

##

- 언어/UI: Swift / SwiftUI / `MenuBarExtra` (style: `.window`)
- 배포 타깃: macOS 14.0+
- 외부 의존성: 없음 (Apple 표준 프레임워크만)
- 데이터: 단일 JSON 파일 — `~/Library/Application Support/Hyperfocus/state.json`
- 진입점: `Hyperfocus/HyperfocusApp.swift`

## 작업 규칙

- 변경이 SPEC을 벗어나면 먼저 SPEC을 업데이트하고 사용자와 합의한 뒤 코드를 고친다.
- 새로운 동작 추가 시 SPEC §10(엣지 케이스 표)과 §11(Out of Scope) 갱신을 잊지 않는다.
- 아키텍처 변경(레이어 책임/의존 방향/저장 포맷)은 architecture.md를 함께 수정한다.
- 단계 진행은 plan.md의 Phase 순서를 따르고, 각 Phase 끝에 명시된 검증을 통과한 후 다음으로 넘어간다.

## 사용자 컨텍스트 (중요)

사용자는 일반 개발 경험은 있지만 **Swift / SwiftUI는 처음**이다. 따라서 디버깅이나
유지보수를 사용자에게 떠넘길 수 없다. 다음 회차 Claude(또는 현재의 너)가 코드만 보고
즉시 상황을 파악할 수 있도록 만들어야 한다. "동작은 하는데 왜 그런지 모르는 코드"는
이 프로젝트에서 명백히 실패다.

### 자기 진단 가능한 코드를 만들 것

- **단위 테스트로 검증한다.** 사용자에게 "한번 눌러봐 주세요"로 검증을 요구하지 않는다.
  도메인 로직(Stores, 합산, 영속화)은 반드시 단위 테스트로 회귀를 잡는다.
- **상태 변화에 로깅을 남긴다.** `os.Logger`로 시작/일시정지/리셋/슬립/저장/로드 이벤트를
  찍어 `Console.app`에서 추적 가능하게 한다. 사용자에게 print 결과를 물어볼 일이 없도록.
- **에러를 삼키지 않는다.** 파일 I/O 실패, 디코딩 실패 등은 로그에 명확한 메시지로 남기고
  가능하면 fallback 동작을 명시한다.

### 차가운 상태에서 읽기 쉬운 코드를 만들 것

- 한 파일은 한 가지 책임만. architecture.md의 폴더 트리 단위를 무너뜨리지 않는다.
- 영리한 Swift보다 평범한 Swift. trailing closure 중첩, 과도한 generic, 매크로 남용 금지.
- 타입 이름과 함수 이름이 그 자체로 의도를 설명하도록 짓는다 (`pause()`가 무엇을 하는지
  주석 없이도 알 수 있게).
- 주석은 **WHY**만. 코드를 읽으면 알 수 있는 WHAT은 쓰지 않는다.
- 비명시적 SwiftUI 매직(`@Environment` 키 자동 매칭 등) 사용 시 등록 지점을 한곳에 모은다.

### "돌아만 가는" 상태에서 멈추지 말 것

- 각 Phase의 verification을 통과했더라도, 다음 회차가 코드를 열었을 때 막히지 않을지
  스스로 한 번 더 검토한다 (네이밍, 파일 위치, 책임 경계).
- 임시 변수/실험 코드/주석 처리된 블록을 남기지 않는다.
- 빌드 경고는 항상 0개로 유지한다.
